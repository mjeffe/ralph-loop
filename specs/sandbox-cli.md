# Sandbox CLI — Lifecycle Commands for Sandbox Management

## Implementation Order

This spec **must be implemented before** `sandbox-setup-prompt.md`. It modifies
the `ralph` script to add the `sandbox` subcommand and all lifecycle functions.
Because ralph-loop uses ralph itself during development, changes to the `ralph`
script must land in a stable state before the next session adds the prompt template
(which is a new file with no risk to the running script).

The split:
- **This spec** adds CLI plumbing: subcommand routing, lifecycle functions (`up`,
  `down`, `reset`, `shell`, `status`, `setup`), installer/updater changes.
- **`sandbox-setup-prompt.md`** adds the prompt template that `sandbox setup`
  invokes. It creates a new file (`prompts/sandbox-setup.md`) and does not modify
  the `ralph` script.

## Overview

The sandbox provides an isolated, all-in-one Docker container where AI coding agents
can execute safely. The container gets its own copy of the codebase (via git clone),
its own services (database, mail, etc.), and only the bare minimum credentials. Code
changes flow in and out exclusively through git.

The sandbox is an optional feature. Projects that don't need containerized execution
continue to run ralph on the host unchanged.

## Design Principles

1. **Ralph remains project-agnostic.** Ralph ships no project-specific runtime
   knowledge. The sandbox Dockerfile, entrypoint, and service configuration are
   generated per-project by an agent via `ralph sandbox setup`.
2. **Ralph has no runtime awareness of the sandbox.** Inside the container, `ralph`
   runs identically to the host. There are no config flags, no conditional agent
   invocation, no sandbox-specific code paths in the loop.
3. **Lifecycle only.** Ralph's sandbox support is limited to lifecycle commands
   (`up`, `down`, `reset`, `setup`, `shell`, `status`) that wrap docker compose.
   These commands only run on the host.
4. **Everything under `.ralph/`.** The sandbox directory lives at `.ralph/sandbox/`,
   consistent with ralph's self-contained installation model.
5. **Git-mediated code flow.** The container's codebase lives on a Docker volume,
   not shared with the host filesystem. Code enters and exits via git push/pull.
   If the agent corrupts the codebase, `ralph sandbox reset` recovers by deleting
   the volume and re-cloning.

## Directory Structure

After `ralph sandbox setup` generates the project-specific files:

```
.ralph/
├── sandbox/
│   ├── docker-compose.yml    # project-specific (generated by setup, never updated by ralph update)
│   ├── Dockerfile            # project-specific (generated by setup, never updated by ralph update)
│   ├── entrypoint.sh         # project-specific (generated by setup, never updated by ralph update)
│   ├── .env                  # gitignored — contains tokens (GITHUB_TOKEN, AMP_API_KEY, etc.)
│   └── .env.example          # committed — template for .env
├── prompts/
│   └── sandbox-setup.md      # managed upstream file — prompt template for setup agent
├── ...
```

### File ownership

| File | Owner | Updated by `ralph update`? |
|---|---|---|
| `prompts/sandbox-setup.md` | upstream (managed) | Yes (checksum-tracked via `.manifest`) |
| `sandbox/docker-compose.yml` | project | No — generated once by `ralph sandbox setup` |
| `sandbox/Dockerfile` | project | No — generated once by `ralph sandbox setup` |
| `sandbox/entrypoint.sh` | project | No — generated once by `ralph sandbox setup` |
| `sandbox/.env.example` | project | No — generated once by `ralph sandbox setup` |
| `sandbox/.env` | user | No — gitignored, never committed |

## CLI Commands

All sandbox commands are subcommands of `ralph sandbox`. They are **host-only** — if
`SANDBOX=1` is set in the environment (indicating execution inside the container),
ralph prints an error and exits.

```
ralph sandbox setup           Generate sandbox files by prompting the agent
ralph sandbox up              Start the sandbox container (build if needed)
ralph sandbox down            Stop the sandbox container
ralph sandbox reset           Re-clone codebase (preserves database volumes)
ralph sandbox reset --all     Delete all volumes (codebase + database) and restart
ralph sandbox shell           Open a bash shell inside the container
ralph sandbox status          Show sandbox container status and service health
```

### Detection of execution context

The `docker-compose.yml` sets `SANDBOX=1` in the container's environment. The
`ralph` script checks this before executing any sandbox subcommand:

```bash
sandbox)
    if [[ "${SANDBOX:-}" == "1" ]]; then
        echo "Error: already inside the sandbox. Run ralph commands directly." >&2
        exit 1
    fi
    # ... handle subcommands
    ;;
```

Inside the container, users run ralph normally (`ralph plan`, `ralph build`, etc.)
with no sandbox prefix.

## `ralph sandbox setup` — Agent-Generated Scaffolding

### Purpose

Rather than shipping stack-specific Dockerfile templates, ralph prompts the
configured agent to scan the project and generate appropriate sandbox files. This
keeps ralph project-agnostic while producing tailored containers.

### Behavior

1. Verify `.ralph/sandbox/` does not already contain generated files (Dockerfile,
   docker-compose.yml, entrypoint.sh). If they exist, print a message and exit
   (the user should delete them manually or use a `--force` flag to regenerate).
2. Create `.ralph/sandbox/` directory if it does not exist.
3. Invoke the configured agent with the `prompts/sandbox-setup.md` prompt template.
   This uses the same `agent_invoke` mechanism as plan/build modes — the configured
   agent script (`agents/${AGENT}.sh`) handles invocation.
4. The agent analyzes the project and creates the sandbox files.
5. The agent commits the generated files.

## `ralph sandbox up`

```bash
sandbox_up() {
    local compose_file="$RALPH_DIR/sandbox/docker-compose.yml"
    if [[ ! -f "$compose_file" ]]; then
        echo "Sandbox not configured. Run 'ralph sandbox setup' first." >&2
        exit 1
    fi
    docker compose -f "$compose_file" up -d --build "$@"
    echo ""
    echo "Sandbox is starting. First build may take several minutes."
    echo "Use 'ralph sandbox shell' to connect once ready."
}
```

## `ralph sandbox down`

```bash
sandbox_down() {
    docker compose -f "$RALPH_DIR/sandbox/docker-compose.yml" down "$@"
}
```

## `ralph sandbox reset`

By default, deletes the codebase volume (re-clone from git on next start) but
preserves database volumes. With `--all`, deletes all volumes including database
data. The user is prompted for confirmation.

```bash
sandbox_reset() {
    local compose_file="$RALPH_DIR/sandbox/docker-compose.yml"
    local reset_all=false

    if [[ "${1:-}" == "--all" ]]; then
        reset_all=true
    fi

    local project_name
    project_name=$(docker compose -f "$compose_file" config --format json \
        | jq -r '.name // empty' 2>/dev/null)

    if [[ -z "$project_name" ]]; then
        echo "Error: could not determine compose project name." >&2
        exit 1
    fi

    if [[ "$reset_all" == true ]]; then
        echo "This will delete ALL sandbox data (codebase AND database)."
    else
        echo "This will delete the sandbox codebase and re-clone from git."
        echo "Database data will be preserved."
    fi
    read -p "Continue? [y/N] " -n 1 -r
    echo
    if [[ $REPLY =~ ^[Yy]$ ]]; then
        if [[ "$reset_all" == true ]]; then
            docker compose -f "$compose_file" down -v
        else
            docker compose -f "$compose_file" down
            docker volume rm "${project_name}_sandbox-codebase" 2>/dev/null || true
        fi
        docker compose -f "$compose_file" up -d --build
        echo "Sandbox reset complete."
    fi
}
```

## `ralph sandbox shell`

```bash
sandbox_shell() {
    local container_name
    container_name=$(sandbox_container_name)
    docker exec -it -u ralph -w /var/www/html "$container_name" bash
}
```

The `sandbox_container_name` helper reads `container_name` from the compose file
or derives it from the project name.

Note: the workdir (`/var/www/html`) should match the `WORKDIR` in the generated
Dockerfile. The setup agent should use a conventional path — `/var/www/html` for
PHP projects, `/app` for most others.

## `ralph sandbox status`

Prints whether the container is running and the health of services inside it.

## Changes to `ralph` Script

Add a `sandbox` case to the main argument parser:

```bash
sandbox)
    if [[ "${SANDBOX:-}" == "1" ]]; then
        echo "Error: already inside the sandbox. Run ralph commands directly." >&2
        exit 1
    fi
    shift
    case "${1:-}" in
        setup)  shift; sandbox_setup "$@" ;;
        up)     shift; sandbox_up "$@" ;;
        down)   shift; sandbox_down "$@" ;;
        reset)  shift; sandbox_reset "$@" ;;
        shell)  shift; sandbox_shell "$@" ;;
        status) shift; sandbox_status "$@" ;;
        *)
            echo "Usage: ralph sandbox <setup|up|down|reset|shell|status>"
            exit 1
            ;;
    esac
    ;;
```

The `sandbox_setup` function:

```bash
sandbox_setup() {
    local sandbox_dir="$RALPH_DIR/sandbox"
    local prompt_template="$RALPH_DIR/prompts/sandbox-setup.md"

    if [[ ! -f "$prompt_template" ]]; then
        echo "Error: sandbox setup prompt not found: $prompt_template" >&2
        exit 1
    fi

    if [[ -f "$sandbox_dir/Dockerfile" ]]; then
        echo "Sandbox files already exist in $sandbox_dir/"
        echo "Delete them first or use --force to regenerate."
        exit 1
    fi

    mkdir -p "$sandbox_dir"

    # Use the same agent invocation as plan/build modes
    local tmp_prompt
    tmp_prompt=$(mktemp)
    prepare_prompt "$prompt_template" "$tmp_prompt"
    agent_invoke "$tmp_prompt" | agent_format_display
    rm -f "$tmp_prompt"
}
```

## Installer Changes (`install.sh`)

The installer adds:

1. `prompts/sandbox-setup.md` to the managed files list (fetched from upstream,
   tracked in `.manifest`).
2. Creates `.ralph/sandbox/` directory.
3. Adds `sandbox/.env` to `.ralph/.gitignore`.

The installer does **not** run `ralph sandbox setup` automatically — that requires
an agent and API key, which may not be configured at install time.

## Updater Changes (`update.sh`)

Add `prompts/sandbox-setup.md` to the `MANAGED_FILES` array. No other changes.
The generated sandbox files (`Dockerfile`, `docker-compose.yml`, `entrypoint.sh`,
`.env.example`) are project-owned and never touched by the updater.

## Typical Workflow

```bash
# 1. Install ralph into a project (existing flow)
curl -sSL https://raw.githubusercontent.com/mjeffe/ralph-loop/main/install.sh | bash

# 2. Generate sandbox files
.ralph/ralph sandbox setup

# 3. Review generated files, adjust if needed
cat .ralph/sandbox/Dockerfile
# (make edits if necessary)

# 4. Create .env with tokens
cp .ralph/sandbox/.env.example .ralph/sandbox/.env
# (fill in GITHUB_TOKEN and AMP_API_KEY)

# 5. Start the sandbox
.ralph/ralph sandbox up

# 6. Shell in and run ralph normally
.ralph/ralph sandbox shell
# Inside container:
.ralph/ralph plan
.ralph/ralph build 10

# 7. When done
exit
.ralph/ralph sandbox down
```

## Non-Goals

- **Running ralph on the host with agent exec'd into container.** The user shells
  into the container and runs ralph directly. No remote execution.
- **Hot-reloading host files into container.** Code flows through git only.
- **Multi-container sandbox.** The sandbox is always a single all-in-one container.
- **Production parity.** The sandbox optimizes for agent development, not production
  deployment.
- **Automatic sandbox startup before ralph plan/build.** The user manages the
  sandbox lifecycle manually.
