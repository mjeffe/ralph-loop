RETROSPECTIVE
=============

After a plan+build cycle, review what happened and improve your inputs
for the next cycle. The retro is human-driven, but two shipped ad-hoc
prompts handle the mechanical parts so you can focus on judgment.

THE THREE-STAGE WORKFLOW
------------------------

  1. ANALYZE (automated)
     ralph prompt .ralph/prompts/adhoc-retro-analyze.md

     The agent reads session logs, the implementation plan, git history,
     AGENTS.md, and specs from the most recent cycle. It produces a
     structured report at .ralph/retro-report.md ranking the top issues
     by wasted effort and proposing fixes categorized as AGENTS.md / spec
     / prompt. The agent does NOT apply any fixes.

     Most retros complete in 1–2 iterations. The example above omits an
     iteration cap to let ralph's default (10) apply — this gives the
     agent room to continue if the cycle's inputs are large enough that
     a single iteration cannot finish. The agent self-decides whether
     it needs to continue and uses TODO markers in the report itself
     for resumption.

  2. DISCUSS (interactive)
     Open an interactive session with your preferred agent and use the
     prompt below to pressure-test the report and draft the actual edits.
     This is where human judgment lives — deciding rankings, scope, and
     timing.

  3. SHARE (optional, automated)
     ralph prompt .ralph/prompts/adhoc-retro-feedback.md

     If you want to share structural feedback with the ralph-loop project,
     this prompt sanitizes .ralph/retro-report.md and writes the result to
     .ralph/retro-feedback.md, ready to paste as a GitHub issue body. All
     project-specific details are stripped out.

     Requires .ralph/retro-report.md to exist (run stage 1 first). If the
     report is missing or incomplete the prompt exits with an error
     message — it does not re-analyze the cycle.

ARTIFACTS ARE TRANSIENT
-----------------------

  Both .ralph/retro-report.md and .ralph/retro-feedback.md are gitignored
  and never committed. Each retro overwrites the previous one. If you
  want to keep an old retro for reference, rename it before running again.

WHEN TO DO A RETRO
------------------
After every significant plan+build cycle, especially when:
  - Build agents struggled, blocked frequently, or triggered REPLAN
  - Tasks took more iterations than expected
  - You see repeated patterns in agent mistakes
  - The plan needed manual editing mid-build

WHAT THE ANALYSIS COVERS
------------------------

The adhoc-retro-analyze prompt examines five dimensions:

  1. Iteration economics — iterations per task, retries, agent failures,
     REPLAN signals; tasks that consumed disproportionate iterations.

  2. Plan quality — blocked tasks, "Assumption / Spec gap:" notes,
     "Conflict:" notes, tasks added during build that could have been
     planned, task sizing.

  3. Git history — revert commits, fix-up commits, unclear messages,
     unrelated changes committed together.

  4. AGENTS.md effectiveness — wrong test/build/lint commands, missing
     conventions, environment-specific issues that should have been
     prevented.

  5. Spec quality — misinterpretations, unclear verification criteria,
     ambiguities that led to guessing.

COMMON FAILURE PATTERNS
-----------------------

  Wrong test commands     → Update AGENTS.md with exact commands and flags
  Unclear "done" state    → Add a Verify section to specs with concrete checks
  Agent scope creep       → Tighten task descriptions in specs, add constraints
  Repeated mistakes       → Add a "Pitfalls" or "Do NOT" section to AGENTS.md
  Bad task sizing         → Review task sizing heuristics in your specs;
                            add examples of good vs bad decomposition
  Spec ambiguity          → Add concrete examples, explicit edge cases,
                            sample inputs/outputs
  Missing prerequisites   → Add dependency/environment setup to AGENTS.md
  Stale plan              → Delete implementation_plan.md and re-plan more often

WHERE TO APPLY FIXES
--------------------

  AGENTS.md               Operational knowledge: commands, environment, conventions,
                          pitfalls. This is what agents read every iteration — keep
                          it concise and accurate.

  Specs (specs/*.md)      Behavioral definitions: what the system should do, how to
                          verify it works. Tighten ambiguous areas, add Verify blocks,
                          add examples.

  Prompts (prompts/*.md)  Structural changes to how agents approach planning or
                          building. Change these rarely and carefully — they affect
                          all projects using ralph.

INTERACTIVE DISCUSSION PROMPT
-----------------------------

After the analyze prompt produces .ralph/retro-report.md, start an
interactive agent session and paste the prompt below. The session is for
exploring findings, pressure-testing rankings, and drafting the actual
edits — not for redoing the analysis.

  --- copy from here ---

  I just ran ralph's automated retro analysis. The report is at
  .ralph/retro-report.md (or retro-report.md). Read it along with the
  session logs in .ralph/logs/, .ralph/implementation_plan.md, and the
  recent git log it cites, then help me:

  1. Pressure-test the top issues — are the rankings right? Anything
     missed? Any "issue" that's actually fine?
  2. For each proposed fix, decide AGENTS.md vs spec vs prompt — and
     whether to apply it now or defer.
  3. Draft the actual edits for the fixes we agree to apply now.
  4. Highlight any patterns I should watch for in the next cycle.

  Ask me before making any file changes.

  --- end copy ---

RETRO CHECKLIST
---------------

  [ ] Run: ralph prompt .ralph/prompts/adhoc-retro-analyze.md
  [ ] Review .ralph/retro-report.md
  [ ] Open an interactive session with the discussion prompt above
  [ ] Apply the agreed-on fixes to AGENTS.md, specs, and/or prompts
  [ ] Commit the improvements before the next plan+build cycle
  [ ] (Optional) ralph prompt .ralph/prompts/adhoc-retro-feedback.md
  [ ] (Optional) Paste .ralph/retro-feedback.md into a ralph-loop GitHub issue

BUILDING INSTITUTIONAL KNOWLEDGE
---------------------------------
The retro isn't just about fixing past issues — it's about preventing
future ones.

  AGENTS.md as accumulated wisdom:
    Every issue you fix twice should become an AGENTS.md entry. It's read
    by every build iteration, making it your "don't do this" memory:
    known pitfalls, naming conventions, architectural constraints, exact
    test commands.

  Spec refinement as continuous practice:
    When you find issues, update the spec — not just the code. If ralph
    ever rebuilds (replan), the fix is captured in the spec. Review
    "Assumption / Spec gap:" notes in the plan — they show exactly where
    specs were ambiguous.

  Pattern recognition:
    Track which categories of issues keep appearing. Common patterns:
    - Same type of edge case missed → add to spec template
    - Agent keeps using wrong approach → add constraint to AGENTS.md
    - Verification too vague → tighten Verify blocks in specs

THE LETTING-GO CHECKLIST
-------------------------
When you catch yourself doing work manually, ask:

  [ ] Could I describe this as a spec and let ralph build it?
  [ ] Could I use ralph prompt for this?
  [ ] Am I pre-diagnosing? Give the agent the symptom, not the diagnosis.
  [ ] Am I writing a spec from scratch? Have the agent draft it first.
  [ ] Have I seen this issue before? Add it to AGENTS.md.
  [ ] Am I testing something that should have an automated test?
      Add test requirements to the spec.

The goal isn't to remove yourself — it's to move from doing the work to
reviewing the work, and eventually to spot-checking the work.
